KEY FEATURES DEMONSTRATED
================================================================================

1. GENERIC HTTP HANDLERS
   ✓ Single set of handlers serve all resources
   ✓ No hardcoded routes
   ✓ Resource type determined at runtime

2. DYNAMIC ROUTING
   ✓ Resource lookup from registry on every request
   ✓ Uses read-lock for fast concurrent lookups
   ✓ Write-lock only for registration (rare)

3. RESOURCE REGISTRY
   ✓ Thread-safe Registry with RWMutex
   ✓ Register resources at runtime
   ✓ Unregister resources at runtime
   ✓ Immediate availability (no restart)

4. TYPE FACTORY PATTERN
   ✓ Scheme creates objects by name
   ✓ No type imports in framework
   ✓ Works with any Go struct

5. CRD SYSTEM
   ✓ Define resources via JSON POST
   ✓ Validate CRD definitions
   ✓ Automatic Resource/Scheme registration
   ✓ Delete CRDs and remove resources

6. API DISCOVERY
   ✓ Kubernetes-style /api endpoint
   ✓ /apis endpoint for groups
   ✓ /apis/{group}/{version} for resources
   ✓ Dynamic updates as CRDs change

7. GENERIC OBJECTS
   ✓ DynamicObject holds any JSON
   ✓ No compiled Go structs required
   ✓ Flexible schema support
   ✓ Metadata handling (Kubernetes-like)

8. DISCOVERY-BASED CLIENT
   ✓ apitcl learns resources from server
   ✓ No hardcoded resource names
   ✓ Works with any resource type
   ✓ YAML and JSON support

HOW TO USE
================================================================================

QUICK START (5 minutes):

1. Build:
   go build -o server ./cmd/server
   go build -o apitcl ./cmd/apitcl

2. Terminal 1 - Start server:
   ./server

3. Terminal 2 - Run demo:
   ./apitcl api-resources
   ./apitcl apply -f examples/invoice-crd.yaml
   ./apitcl api-resources        # Notice invoices appears!
   ./apitcl create -f examples/invoice-1.json
   ./apitcl get invoices
   ./apitcl delete crd invoices.example.io
   ./apitcl api-resources        # Notice invoices is gone!

================================================================================
EDUCATIONAL VALUE
================================================================================

This project clearly demonstrates:

1. How Kubernetes achieves runtime extensibility
   • No API server restart needed for new CRDs
   • Generic HTTP handlers
   • Dynamic resource registry
   • Thread-safe concurrent access

2. Key design patterns
   • Factory pattern (Scheme)
   • Registry pattern (Resource management)
   • Interface-based design (Resource, Storage)
   • Thread-safety (RWMutex)

3. Architectural decisions
   • Why routes are generic, not hardcoded
   • Why resources are looked up per-request
   • Why handlers work with interfaces
   • Why objects are generic, not typed

4. Performance characteristics
   • Read locks don't block each other
   • Write locks are rare
   • Lookup is O(1) hashtable access
   • Network latency dominates

5. Extensibility without modification
   • Add CRD = add resource
   • No code changes needed
   • No recompilation
   • No restart
   • Happens at runtime
